# PRD: Apache Guacamole 기반 웹 원격접속 플랫폼

## 1. 개요

본 문서는 Apache Guacamole을 기반으로 서버, 데스크톱, VM, 내부 업무 시스템에 웹브라우저로 원격 접속할 수 있는 플랫폼을 구축하기 위한 제품 요구사항 문서이다.

목표는 별도 클라이언트 설치 없이 HTTPS 웹브라우저만으로 RDP, SSH, VNC 접속을 제공하고, 사용자별 권한 제어, 인증, 감사 로그, 세션 관리가 가능한 중앙 원격접속 게이트웨이를 운영하는 것이다.

## 2. 배경 및 문제 정의

현재 원격 서버 접속은 VPN, SSH 클라이언트, RDP 클라이언트, 개별 계정 관리 등으로 분산되어 운영될 가능성이 높다. 이 방식은 다음 문제가 있다.

- 사용자별 접속 권한과 이력 관리가 어렵다.
- 접속 대상 서버의 포트가 외부 또는 여러 네트워크 구간에 노출될 수 있다.
- 접속 도구가 사용자 PC 환경에 의존한다.
- 퇴사자, 외주 인력, 임시 계정의 접근 통제가 번거롭다.
- 장애 대응 또는 작업 이력 추적 시 증적 확보가 어렵다.

Apache Guacamole 기반 플랫폼은 원격접속을 단일 웹 게이트웨이로 통합하여 보안, 운영, 사용성을 개선한다.

## 3. 목표

### 3.1 제품 목표

- 웹브라우저만으로 서버에 원격 접속할 수 있는 환경 제공
- RDP, SSH, VNC 접속을 하나의 포털에서 통합 관리
- 사용자/그룹별 접속 대상 권한 제어
- 중앙 인증과 2차 인증 적용
- 접속 이력, 실패 이력, 세션 기록 관리
- Docker 기반으로 재현 가능한 배포 환경 구성

### 3.2 비즈니스/운영 목표

- 원격접속 운영 비용 절감
- 사용자 온보딩/오프보딩 절차 단순화
- 서버 직접 노출 최소화
- 장애 대응 및 보안 감사 대응력 향상
- 향후 자체 포털, 승인 워크플로우, 고객사별 격리 구조로 확장 가능한 기반 확보

## 4. 범위

### 4.1 MVP 범위

- Apache Guacamole 서버 구축
- 웹브라우저 접속 URL 제공
- 관리자 계정 및 일반 사용자 계정 관리
- RDP, SSH 접속 대상 등록
- 사용자별 접속 권한 설정
- PostgreSQL 또는 MySQL 기반 설정 저장소 구성
- HTTPS 적용
- 기본 접속 로그 확인
- Docker Compose 기반 배포
- 백업/복구 절차 문서화

### 4.2 2차 범위

- LDAP, Active Directory, OIDC, SAML 중 하나 이상의 중앙 인증 연동
- TOTP 기반 2차 인증
- 세션 녹화 및 재생
- 사용자 그룹별 접속 정책
- 접속 시간 제한
- IP 허용 목록
- 관리자 감사 로그
- 모니터링 및 알림
- 고가용성 구성 검토

### 4.3 제외 범위

- 자체 RDP/VNC/SSH 프로토콜 구현
- 자체 원격 제어 엔진 개발
- Guacamole 소스코드 대규모 수정
- 모바일 앱 개발
- 상용 원격지원 도구 수준의 실시간 고객지원 기능
- 파일 보안 DLP, 화면 워터마크, 행위 분석 등 고급 보안 제품 기능

## 5. 사용자 및 권한

### 5.1 사용자 유형

| 사용자 | 설명 | 주요 기능 |
|---|---|---|
| 시스템 관리자 | 플랫폼 전체를 관리하는 운영자 | 사용자, 그룹, 접속 대상, 정책 관리 |
| 일반 사용자 | 허가된 서버에 접속하는 사용자 | 웹 로그인, 원격접속, 세션 종료 |
| 감사 담당자 | 접속 이력과 세션 기록을 확인하는 사용자 | 로그 조회, 기록 검토 |
| 외부 협력사 | 제한된 기간 동안 일부 서버에 접속하는 사용자 | 제한된 서버 접속, 기간 제한 |

### 5.2 권한 모델

- 사용자는 하나 이상의 그룹에 속할 수 있다.
- 접속 대상은 사용자 또는 그룹 단위로 할당한다.
- 관리자 권한은 최소 인원에게만 부여한다.
- 외부 협력사 계정은 만료일을 필수로 설정한다.
- 운영 서버, 개발 서버, 고객사 서버는 그룹 단위로 분리한다.

## 6. 핵심 사용자 시나리오

### 6.1 일반 사용자 원격접속

1. 사용자가 웹브라우저로 원격접속 포털에 접속한다.
2. 계정과 2차 인증으로 로그인한다.
3. 본인에게 허용된 접속 대상 목록을 확인한다.
4. 접속 대상을 선택한다.
5. 브라우저 안에서 RDP 또는 SSH 세션을 사용한다.
6. 작업 종료 후 세션을 종료한다.
7. 시스템은 접속 시작/종료 시간을 기록한다.

### 6.2 관리자 접속 대상 등록

1. 관리자가 관리자 콘솔에 로그인한다.
2. 신규 서버 접속 정보를 등록한다.
3. 프로토콜, 호스트, 포트, 인증 방식을 설정한다.
4. 접속 가능 사용자 또는 그룹을 지정한다.
5. 테스트 접속을 수행한다.
6. 사용자에게 접속 권한을 안내한다.

### 6.3 외부 협력사 임시 접근

1. 관리자가 협력사 계정을 생성한다.
2. 계정 만료일과 접속 가능 시간대를 설정한다.
3. 특정 서버 또는 그룹에만 접속 권한을 부여한다.
4. 작업 완료 후 계정을 비활성화하거나 자동 만료 처리한다.
5. 필요 시 접속 이력과 세션 기록을 검토한다.

## 7. 기능 요구사항

### 7.1 인증

- 사용자는 웹 로그인으로 접속해야 한다.
- MVP에서는 Guacamole 기본 인증 또는 데이터베이스 인증을 사용한다.
- 운영 환경에서는 중앙 인증 연동을 우선 검토한다.
- 2차 인증은 운영 오픈 전 필수 적용을 목표로 한다.
- 관리자 계정은 일반 사용자 계정과 분리한다.

### 7.2 접속 대상 관리

- 관리자는 RDP, SSH, VNC 접속 대상을 등록할 수 있어야 한다.
- 접속 대상별 이름, 설명, 프로토콜, 호스트, 포트, 계정 정보를 설정할 수 있어야 한다.
- 접속 정보는 사용자에게 직접 노출되지 않아야 한다.
- 접속 대상은 그룹별로 분류할 수 있어야 한다.

### 7.3 권한 관리

- 사용자별 접속 가능 대상을 설정할 수 있어야 한다.
- 그룹별 접속 가능 대상을 설정할 수 있어야 한다.
- 외부 사용자에게 기간 제한 권한을 부여할 수 있어야 한다.
- 퇴사자 또는 프로젝트 종료 사용자는 즉시 비활성화할 수 있어야 한다.

### 7.4 원격접속

- 사용자는 브라우저에서 RDP 세션을 사용할 수 있어야 한다.
- 사용자는 브라우저에서 SSH 터미널을 사용할 수 있어야 한다.
- 필요 시 VNC 접속을 사용할 수 있어야 한다.
- 세션은 HTTPS 기반으로 제공되어야 한다.
- 네트워크 단절 또는 브라우저 종료 시 세션 정리 정책이 적용되어야 한다.

### 7.5 로그 및 감사

- 로그인 성공/실패 이력을 기록해야 한다.
- 원격접속 시작/종료 이력을 기록해야 한다.
- 사용자, 접속 대상, 접속 시각, 종료 시각, 접속 결과를 확인할 수 있어야 한다.
- 관리자 변경 작업 이력을 남기는 방안을 검토한다.
- 세션 녹화는 2차 범위로 포함한다.

### 7.6 운영 관리

- Docker Compose로 서비스 시작/중지/재시작이 가능해야 한다.
- 설정 데이터베이스는 백업 가능해야 한다.
- 장애 발생 시 복구 절차가 문서화되어야 한다.
- 서비스 상태 확인 방법을 제공해야 한다.

## 8. 비기능 요구사항

### 8.1 보안

- 모든 외부 접속은 HTTPS를 사용해야 한다.
- Guacamole 서버는 접속 대상 서버와 같은 내부망 또는 허용된 네트워크 구간에 위치해야 한다.
- RDP, SSH, VNC 포트는 인터넷에 직접 노출하지 않는다.
- 관리자 콘솔 접근은 IP 제한 또는 별도 보안 정책을 적용한다.
- 기본 관리자 계정은 초기 구축 후 즉시 변경하거나 비활성화한다.
- 비밀번호 정책과 2차 인증을 적용한다.
- 정기적으로 Guacamole 및 컨테이너 이미지를 업데이트한다.

### 8.2 성능

- MVP 기준 동시 접속 10명 이상을 지원한다.
- 2차 목표로 동시 접속 50명 이상을 지원할 수 있는 확장 구조를 검토한다.
- RDP 화면 응답성은 일반 서버 관리 작업에 지장이 없어야 한다.
- 파일 전송, 오디오, 고해상도 그래픽 사용은 성능 영향을 검토한 뒤 제한할 수 있다.

### 8.3 가용성

- MVP는 단일 인스턴스 구성을 허용한다.
- 운영 중요도가 높아질 경우 DB 백업, 컨테이너 재시작 정책, 프록시 이중화, Guacamole 다중 인스턴스를 검토한다.
- 장애 시 복구 목표 시간은 MVP 기준 1시간 이내로 한다.

### 8.4 호환성

- 최신 Chrome, Edge, Firefox 브라우저를 지원한다.
- Windows 서버 RDP 접속을 지원한다.
- Linux 서버 SSH 접속을 지원한다.
- 필요 시 Linux GUI 환경의 VNC/RDP 접속을 지원한다.

## 9. 권장 아키텍처

### 9.1 MVP 구성

```text
User Browser
    |
    | HTTPS
    v
Reverse Proxy / TLS
    |
    v
Apache Guacamole Web
    |
    v
guacd
    |
    +--> RDP Target Servers
    +--> SSH Target Servers
    +--> VNC Target Servers

Guacamole Web
    |
    v
PostgreSQL or MySQL
```

### 9.2 구성 요소

| 구성 요소 | 역할 |
|---|---|
| Reverse Proxy | HTTPS 종료, 도메인 라우팅, 보안 헤더 적용 |
| Guacamole Web | 사용자 UI, 인증, 권한, 접속 목록 제공 |
| guacd | RDP/SSH/VNC 프로토콜 중계 |
| PostgreSQL/MySQL | 사용자, 권한, 접속 설정 저장 |
| Backup Storage | DB 백업, 설정 백업 저장 |
| Monitoring | 서비스 상태, 로그, 리소스 모니터링 |

## 10. 데이터 및 백업

- 데이터베이스에는 사용자, 그룹, 접속 대상, 권한 설정이 저장된다.
- 접속 대상 비밀번호 저장 방식은 운영 보안 정책에 맞게 결정한다.
- DB 백업은 최소 1일 1회 수행한다.
- 백업 파일은 운영 서버와 분리된 위치에 보관한다.
- 복구 테스트는 초기 구축 후 1회 이상 수행한다.

## 11. 배포 요구사항

### 11.1 환경

- Linux 서버 또는 Linux VM
- Docker, Docker Compose
- PostgreSQL 또는 MySQL
- HTTPS 인증서
- 내부 접속 대상 서버와의 네트워크 통신 가능

### 11.2 도메인 예시

- 운영: `remote.example.com`
- 관리자 접근: 동일 도메인 내 관리자 메뉴 또는 별도 접근 정책 적용

### 11.3 네트워크 정책

- 외부 사용자는 HTTPS 포트만 접근 가능해야 한다.
- Guacamole 서버는 내부 RDP, SSH, VNC 포트에 접근 가능해야 한다.
- 접속 대상 서버는 Guacamole 서버의 IP만 허용하도록 제한하는 것을 권장한다.

## 12. 보안 정책

- 모든 사용자는 개인 계정을 사용한다.
- 계정 공유를 금지한다.
- 외부 협력사 계정은 만료일을 설정한다.
- 관리자 권한은 승인된 인원에게만 부여한다.
- 접속 로그는 최소 90일 이상 보관한다.
- 중요 서버 접속은 세션 녹화 적용을 검토한다.
- 보안 업데이트는 월 1회 이상 검토한다.
- 긴급 취약점 공지 시 즉시 패치 계획을 수립한다.

| 계정 공유 | 감사 추적 어려움 | 개인 계정 원칙, 로그 감사, 계정 정책 |
| 접속 대상 비밀번호 노출 | 서버 침해 가능성 | 저장 방식 검토, 권한 제한, 주기적 변경 |
| 성능 저하 | 원격 작업 불편 | 동시 접속 기준 설정, 리소스 모니터링 |
| 단일 장애점 | 서비스 중단 | 백업, 재시작 정책, 향후 이중화 |
| 브라우저 호환성 문제 | 사용자 접속 실패 | 표준 브라우저 지정, 사전 테스트 |

## 16. 의사결정 사항

- 데이터베이스는 PostgreSQL과 MySQL 중 무엇을 사용할 것인가?

- 2차 인증을 MVP 필수 범위로 포함할 것인가?
- 외부 접속은 VPN 뒤에서만 허용할 것인가, 인터넷 공개 + 강한 인증으로 운영할 것인가?
- 세션 녹화가 필요한 서버 범위는 어디까지인가?
- 접속 대상 서버의 계정 관리는 공용 계정 방식인가, 개인 계정 방식인가?

## 17. MVP 완료 기준

- 사용자가 웹브라우저로 Guacamole에 로그인할 수 있다.
- 사용자가 허용된 RDP 서버에 접속할 수 있다.
- 사용자가 허용된 SSH 서버에 접속할 수 있다.
- 관리자가 사용자와 접속 대상을 등록할 수 있다.
- 사용자별 또는 그룹별 권한이 적용된다.
- HTTPS가 적용되어 있다.
- 접속 로그를 확인할 수 있다.
- DB 백업과 복구 절차가 문서화되어 있다.
- 기본 보안 설정이 적용되어 있다.

## 18. 권장 결론

MVP는 Apache Guacamole을 직접 수정하지 않고 공식 컨테이너 기반으로 구축한다. 자체 플랫폼 개발은 원격접속 엔진을 새로 만드는 방식이 아니라, Guacamole을 내부 원격접속 게이트웨이로 사용하고 필요 시 별도 포털, 승인 프로세스, 계정 자동화, 감사 대시보드를 주변에 붙이는 방식으로 추진한다.

이 접근은 초기 구축 속도, 보안 안정성, 운영 편의성, 향후 확장성 측면에서 가장 현실적이다.
